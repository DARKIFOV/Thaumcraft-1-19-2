# Thaumcraft Legacy Rebuild v11.62.24

## Research Matrix + Thaumonomicon + Infusion Matrix + Deconstruction + Crystalizer + Advanced Furnace Controller Original Parity

Этап выполнен по исходникам Thaumcraft 4.2.3.5:

- `ResearchManager` и структура `ResearchNoteData`;
- `GuiResearchBrowser`;
- `TileResearchTable`;
- `TileInfusionMatrix`;
- `TileDeconstructionTable`;
- `TileEssentiaCrystalizer`;
- `TileAlchemyFurnaceAdvanced`;
- `TileAlchemyFurnaceAdvancedNozzle`;
- оригинальные рецепты `ConfigRecipes`.

Этот пакет закрывает не один блок, а сразу исследовательский контур, важную ошибку инфузионной матрицы и несколько отсутствовавших алхимических устройств.

# 1. Исследовательская матрица записок

## 1.1. Случайная схема больше не пересоздаётся детерминированно

Предыдущая реализация могла строить одинаковую упрощённую схему для одного исследования. В TC4 новая записка создаётся из мирового потока случайности, а получившаяся матрица после создания остаётся постоянной.

Теперь при создании записки:

- используется случайный `long` из серверного RNG игрока;
- seed сохраняется как `GridSeed`;
- клиент и сервер читают один и тот же seed;
- выгрузка чанка и перезапуск мира не меняют матрицу;
- копии записки сохраняют исходную схему;
- две новые записки одного исследования сложности выше 1 могут отличаться расположением вырезанных пустых ячеек; для сложности 1 исходная геометрия остаётся одинаковой, как в TC4.

## 1.2. Радиус и сложность

Восстановлена формула:

```text
radius = 1 + min(3, complexity)
```

Поддерживаются радиусы 2–4 и до 61 индексированной ячейки.

Для исследований сложности выше 1 удаляется:

```text
complexity × 2 пустых ячейки
```

Удаление повторяет защиту исходного TC4: нельзя вырезать пустую ячейку, если после этого фиксированный якорь исследования окажется практически изолирован.

## 1.3. Якоря аспектов

Обязательные аспекты размещаются по внешнему кольцу:

- с равномерным расстоянием;
- равномерно от начала кольца без фактического поворота; оригинальный `HexUtils` потребляет случайное значение старта, но из-за исторической ошибки не применяет его;
- как фиксированные `TYPE_RESEARCH_ANCHOR`;
- их нельзя удалить обычным кликом;
- размещённые игроком аспекты хранятся отдельно как `TYPE_PLACED`.

## 1.4. Настоящий формат `hexgrid`

Современная индексированная матрица теперь параллельно записывает оригинальный формат TC4:

```text
hexgrid[]
  hexq: byte
  hexr: byte
  type: byte
  aspect: string, если аспект присутствует
```

На верхнем уровне предмета также сохраняются:

```text
key
complete
copies
color
```

Это важно для:

- совместимости со старыми записками;
- проверки q/r-координат;
- копирования предметов между реализациями;
- диагностики через NBT;
- предотвращения потери формы матрицы.

## 1.5. Импорт настоящих записок TC4

Добавлен обратный импорт.

Если предмет содержит только старый верхнеуровневый `hexgrid`, но не содержит современной структуры `ThaumcraftResearchNote`, перенос:

- читает `hexq` и `hexr`;
- находит современный индекс ячейки;
- восстанавливает тип;
- восстанавливает аспект;
- рассчитывает радиус;
- переносит `key`, `complete` и `copies`;
- не генерирует новую схему поверх старой.

## 1.6. Решение матрицы

Сохранены и укреплены:

- проверка связности якорей;
- проверка допустимых связей аспектов;
- требование знания аспектов игроком;
- очистка несвязанных лишних цепочек при завершении;
- синхронизация `complete=true` в обоих форматах NBT;
- процент прогресса для GUI.

# 2. Транзакционное создание записок

Исправлена выдача записки через стол исследования и переносимый путь создания.

Теперь перед операцией сохраняются копии:

- бумаги;
- письменных принадлежностей;
- запаса чернил.

Если любая часть операции завершается ошибкой:

- бумага возвращается;
- письменные принадлежности возвращаются;
- чернила не теряются;
- пустая или повреждённая записка не создаётся.

Новая записка получает seed из серверного RNG только после успешной проверки ресурсов.

# 3. Таумономикон

## 3.1. Настоящий viewport карты

Восстановлена область из `GuiResearchBrowser`:

```text
x = left + 16
y = top + 17
width = 224
height = 196
```

Карта, линии и иконки теперь реально обрезаются framebuffer-scissor, а не только логически проверяются координатами.

Scissor переводится из GUI-координат в реальные framebuffer-пиксели с учётом `guiScale` и переворота оси Y.

## 3.2. Пропорциональное движение фона

Фон вкладки больше не движется произвольным фиксированным шагом.

Используется пропорциональное положение между минимальной и максимальной границей панорамирования с исходными диапазонами текстуры:

```text
288 по X
316 по Y
```

## 3.3. Память позиции для каждой вкладки

Каждая категория хранит собственные:

```text
panX
panY
```

При переходе между Основами, Алхимией, Тауматургией, Големоведением, Древними знаниями и аддонами карта возвращается туда, где пользователь остановился именно в этой вкладке.

## 3.4. Исправлено случайное открытие исследования

Раньше начало перетаскивания на иконке могло открыть исследование при отпускании мыши.

Теперь:

- запоминается нажатая иконка;
- введён порог движения 4 пикселя;
- до прохождения порога действие считается кликом;
- после прохождения порога действие становится перетаскиванием;
- исследование не открывается после панорамирования.

## 3.5. Сохранена существующая большая система страниц

Новая работа не сломала:

- разметку `<BR>`, цвета и форматирование;
- recipe cards;
- подсказки;
- скрытые и concealed-исследования;
- lost и special-вкладки;
- переходы между страницами;
- отображение аспектов, варпа и зависимостей.

# 4. Инфузионная матрица

## 4.1. Удалена неправильная предварительная проверка всей эссенции

Предыдущая версия не позволяла начать инфузию, пока рядом уже не находилась вся стоимость рецепта.

Это отличается от `TileInfusionMatrix.craftingStart` TC4.

Оригинальная матрица:

1. фиксирует катализатор и компоненты;
2. запускает крафт;
3. затем `craftCycle` по одной единице ищет нужную эссенцию;
4. при отсутствии аспекта ждёт и повторяет попытку.

Теперь перенос работает так же.

Можно запустить инфузию и затем подавать эссенцию:

- трубами;
- банками;
- резервуарами;
- эссенционным големом;
- другой алхимической инфраструктурой.

## 4.2. Расширены источники эссенции

Каждый цикл матрица ищет ближайший подходящий источник в исходном радиусе и забирает ровно одну единицу.

Поддержаны:

- обычные банки;
- фильтрованные и void-банки через общую сущность банки;
- резервуары эссенции;
- алембики;
- буферные трубы;
- готовый выход центрифуги;
- продвинутая алхимическая печь.

Источники сортируются:

1. по расстоянию до матрицы;
2. по Y;
3. по X;
4. по Z.

Это исключает зависимость результата от порядка обхода `BlockPos.betweenClosed`.

## 4.3. Сохранена транзакционная блокировка рецепта

Проверены и сохранены:

- точный катализатор с NBT и повреждением;
- конкретный пьедестал каждого летящего компонента;
- порядок компонентов;
- контейнерные предметы;
- XP для инфузионных зачарований;
- instability events;
- симметрия;
- стабилизаторы;
- terminal failure;
- сохранение процесса после перезапуска.

## 5.1. Оригинальный цикл

- один входной слот;
- длительность разбора — 40 тиков;

- пока результат не забран, следующий предмет не разбирается;

## 5.2. Разложение аспектов

```text

Ignis
Aqua
Ordo
Perditio
```

После этого используется исходная проверка:

```text
random.nextInt(80) < суммарный размер первичных аспектов
```

При успехе случайно выбирается один из присутствующих первичных аспектов.

## 5.3. Получение знания

При клике по появившемуся аспекту:

- игрок получает 1 единицу знания в личный пул;
- аспект считается открытым;
- знания синхронизируются клиенту;
- результат очищается;
- стол продолжает работу.

Как в оригинале:

- сверху слот не доступен;
- с боков и снизу предмет можно подавать;
- извлечение поддерживается;
- воронки не обходят блокировку готового знания.

## 5.5. GUI

Используется исходная текстура:

```text

```

Отображаются:

- входной слот;

- полученный аспект;
- интерактивная область получения знания.

# 6. Кристаллизатор эссенции

Добавлены:

```text
EssentiaCrystalizerBlock
EssentiaCrystalizerBlockEntity
EssentiaCrystalizerRenderer
EssentiaCrystalItem
```

## 6.1. Suction

```text
пустой кристаллизатор: 128
с уже принятой эссенцией: 64
при красном камне: 0
```

Устройство принимает эссенцию только со стороны входа.

Прямой источник может быть:

- трубой эссенции;
- обычной или фильтрованной банкой;
- резервуаром;
- алембиком;
- верхним выходом центрифуги;
- сопоставимым выходом продвинутой алхимической печи.

За один цикл извлекается ровно одна единица, без чтения всей сети как общего резервуара.

Жезлом с Shift можно изменить входную сторону.

## 6.2. Цикл кристаллизации

Каждые 5 тиков:

- если устройство пустое — оно пытается втянуть 1 эссенцию;
- если аспект уже принят — прогресс увеличивается минимум на 1;
- дополнительно может быть забрано до 20 vis Terra;
- каждая единица Terra добавляет 2 прогресса.

Максимум:

```text
200 progress
```

## 6.3. Готовый кристалл

Создан функциональный предмет `tc4_crystalessence`.

Он хранит в NBT:

```text
Aspect
```

Имеет:

- цвет аспекта;
- динамическое имя;
- подсказку;
- сохранение после перезапуска;
- отдельную исходную текстуру.

## 6.4. Вывод

Готовый кристалл:

1. пытается войти в соседний Forge `ITEM_HANDLER`;
2. сохраняет остаток при частичной вставке;
3. выбрасывается с выходной стороны, если инвентаря нет или он заполнен.

## 6.5. Красный камень и рендер

- сигнал красного камня полностью останавливает suction и прогресс;
- вращение ускоряется на 0,1 до 20;
- замедляется на 0,2;
- внутри отображается цветной вращающийся кристалл;
- сохранены исходные модельные ресурсы `crystalizer.obj` и `crystalizer.png`.

# 7. Продвинутая алхимическая печь

В предыдущей версии advanced-блок отличался в основном вместимостью 500. Теперь перенесено рабочее ядро `TileAlchemyFurnaceAdvanced`.

## 7.1. Три запаса энергии

```text
Ignis heat:       0–500
Perditio power:   0–500
Aqua power:       0–500
```

Каждые 5 тиков контроллер:

- уменьшает heat на 1;
- запрашивает до 50 cv Ignis;
- запрашивает до 50 cv Perditio;
- запрашивает до 50 cv Aqua.

Поскольку современный узел хранит целые единицы vis по 100 cv, добавлены отдельные внутренние кредиты:

```text
IgnisCreditCv
PerditioCreditCv
AquaCreditCv
```

При получении 100 cv половина используется сейчас, а оставшиеся 50 cv сохраняются для следующего цикла. Энергия не исчезает из-за несовпадения дискретности.

## 7.2. Поглощение предметов

Как в оригинальном блоке, выброшенный предмет обрабатывается при попадании внутрь печи.

Операция выполняется только если:

- прошла задержка прошлого предмета;
- у предмета есть аспекты;
- в хранилище хватает места;
- хватает всех трёх энергий.

## 7.3. Точная стоимость

Для предмета с суммарным размером аспектов `aa`:

```text
heat      -= aa × 2
Perditio  -= aa
Aqua      -= aa
```

## 7.4. Задержка

Восстановлена формула:

```text
5 + max(0, (1 − heat / 500) × 100)
```

Чем холоднее печь после операции, тем дольше она не принимает следующий предмет.

## 7.5. Вместимость и транспорт

- вместимость — 500 эссенции;
- эссенция сохраняется по аспектам;
- состояние энергии и задержки сохраняется в NBT;
- трубы могут забирать готовую эссенцию с горизонтальной стороны;
- выдача выполняется по одной единице;
- при откате транспортной операции эссенция возвращается.

## 7.6. Честное ограничение

Функциональный контроллер перенесён, но внешний многоблок пока адаптирован упрощённо.

Отдельного полного прохода ещё требуют:

- точная конструкция 3×2×3;
- metadata-варианты стенок;
- четыре отдельные сопловые части;
- разрушение всей конструкции при снятии части;
- точная OBJ/GL11-анимация всей печи;
- визуальная зависимость света от heat.

Механика энергии, обработки и хранения уже работает; геометрия и multiblock validation пока не заявляются как 100%.
