# TC4 v11.63.41 — точный ритуал Node in a Jar и перенос профиля узла

## Цель этапа

После закрытия registry fallback и восстановления визуала пьедестала оставалась крупная source-level подмена: `node_jar` существовал как динамически отрисовываемый предмет, но живой узел захватывался упрощённым прямым действием. В Thaumcraft 4 захват выполнялся жезлом через специальную многоблочную конструкцию. v11.63.41 переносит полный цикл: построение, расход Vis, преобразование мира, перенос NBT, предмет/блок банки и освобождение узла.

## Сверка с исходниками TC4 4.2.3.5

Перенос выполнен по `WandManager#createNodeJar`, `fitNodeJar`, `replaceNodeJar`, `TileJarNode`, `ItemJarNode` и ветке узла в `TileJarRenderer`.

Исходная конструкция имеет размер 3×4×3:

- верхний слой — девять любых деревянных плит из семейства `slabWood`;
- три нижних слоя — стекло;
- живой узел заменяет центральное стекло среднего слоя;
- итого расходуется 26 стеклянных блоков и 9 плит.

Активация требует по 70 centivis каждого первичного аспекта: Aer, Terra, Ignis, Aqua, Ordo и Perditio. Современная реализация использует существующую систему cap/equipment discounts жезла и сначала проверяет все шесть стоимостей, поэтому недостаток одного аспекта не оставляет частично списанный Vis.

## Захват профиля

`TC4NodeJarRuntime.capture` теперь:

- копирует текущие аспекты без процентного уменьшения;
- копирует базовые аспекты;
- сохраняет тип и постоянный `NodeId`;
- с вероятностью 75% ослабляет модификатор ровно на одну ступень:
  - Bright → Normal;
  - Normal → Pale;
  - Pale → Fading;
  - Fading → Fading;
- при оставшихся 25% сохраняет модификатор без изменения.

Удалено ранее выдуманное масштабирование аспектов до 75/92 процентов. Поле `PreservationPercent=100` остаётся только явным migration-маркером для новых банок.

## Многоблочная активация

Создан `TC4NodeJarMultiblock`:

- повторяет исходный диапазон поиска относительно кликнутого блока;
- принимает клики по стеклу, плитам и самому узлу;
- использует `BlockTags.WOODEN_SLABS` как современный эквивалент `slabWood`;
- требует обычное стекло, как исходный blueprint;
- после успешной атомарной оплаты удаляет 35 материалов и заменяет узел блоком банки;
- синхронизирует NBT и запускает односекундную анимацию схлопывания.

Проверка ритуала добавлена и в `WandItem#useOn`, и в `AuraNodeBlock#use`. Это необходимо из-за порядка vanilla-взаимодействия: сам блок узла получает `Block#use` раньше, чем жезл получает `Item#useOn`.

## Блок, предмет и освобождение

Добавлены отдельные `NodeJarBlock` и `NodeJarBlockEntity`:

- профиль узла сохраняется при выгрузке чанка;
- update tag и update packet передают данные клиентскому renderer;
- разрушение создаёт один заполненный `node_jar` с исходным профилем;
- повторная установка восстанавливает профиль в BlockEntity;
- применение жезла заменяет банку живым `AuraNodeBlockEntity`;
- освобождённый узел получает `jarred=false` и снова доступен для обычного tapping и пьедестала.

Последний пункт исправляет runtime-ошибку старой адаптации: раньше освобождённый узел оставался логически «в банке» и исключался из систем зарядки.

## Идентификатор узла

В `AuraNodeBlockEntity` добавлен постоянный `NodeId`:

- создаётся для новых узлов;
- сохраняется в world NBT и update tag;
- переносится в банку;
- восстанавливается после размещения и освобождения;
- для старых миров без поля генерируется один раз при загрузке.

## Визуал

Мировой `NodeJarRenderer` повторно использует уже существующий NBT-aware renderer предмета:

- стеклянная банка и крышка;
- тип, модификатор и аспекты содержащегося узла;
- оригинальная формула схлопывания `1 + 2 × remainingSeconds`;
- масштаб от 3.0 до 1.0 за одну секунду;
- расширенный render bounding box на время эффекта.

`NodeJarItem` реализован как `BlockItem` с одним общим экземпляром BEWLR. Пустые старые migration-стеки не размещаются и показывают подсказку о ритуале.

## Runtime-протокол

Добавлены четыре сценария:

1. `aura_nodes.tc4_node_jar_multiblock_capture`;
2. `aura_nodes.tc4_node_jar_profile_preservation`;
3. `aura_nodes.tc4_node_jar_transport_release`;
4. `visuals.tc4_node_jar_collapse_render`.

Манифест расширен с 292 до **296** сценариев. Все новые сценарии имеют статус `NOT_TESTED` до настоящего запуска Forge 1.19.2 под Java 17.

## Ограничения

Статические guards не доказывают:

- фактический порядок client/server hand interaction;
- отсутствие ghost-блоков при сетевой задержке;
- статистическое распределение 75/25 на длинной серии;
- сохранность профиля после реального перезапуска dedicated server;
- корректность translucent/additive rendering на всех GPU;
- совместимость с чужими деревянными плитами, добавленными в vanilla-тег.

JAR не заявляется без успешного `gradlew build` или `compileJava` под Java 17.
