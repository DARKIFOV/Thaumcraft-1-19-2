# TC4 v11.63.65 — Component refund ledger, случайный выбор аспекта, исправление event-table: отчёт о портировании

## Как это было устроено в оригинале (Thaumcraft 4, 1.7.10, `TileInfusionMatrix.craftCycle`, строки ~415–490)

Когда активная инфузия временно не может продолжиться из-за нехватки ресурсов, оригинал **никогда не отменяет крафт** — он ждёт и по чуть-чуть накручивает дестабилизацию каждый цикл. Ключевая семантика, которая была утеряна при портировании:

1. **Семантика AspectList**. В оригинальном `AspectList.reduce` значение при достижении нуля **не удалялось** из мапы — ключ оставался со значением 0. Поэтому `recipeEssentia.visSize() == 0`, но `recipeEssentia.getAspects()` возвращало **весь исходный набор ключей аспектов**, включая обнулённые. Это позволяло компонентной/xp-нехватке выбирать случайный аспект из исходного множества.

2. **Случайный выбор аспекта для refund**. Оригинал в компонентной нехватке выбирал аспект случайно из всего исходного набора: `Aspect as = ingEss[random.nextInt(ingEss.length)]`. Современный порт использовал `InfusionProcessHelper.firstPendingAspect(pendingAspects)` — детерминированный порядок, а не случайный.

3. **Нехватка эссенции** (исправлено в v11.63.64): за один тик пробует слить все pending-аспекты; после полного провала выставляет `checkSurroundings = true`.

4. **Нехватка компонента**: для каждого несовпавшего ингредиента (индекс `a`) бросает `nextInt(1 + a)`; при 0 возвращает 1 ед. случайного аспекта и вложенно бросает `nextInt(50 - instability*2)` на instability-creep.

5. **Event table gate**: оригинал использовал `nextInt(500) < instability` (строгое меньше, а не ≤) с предварительной проверкой `instability > 0`.

## Что было неверно в порте (v11.63.64 и ранее)

- **Component refund недостижим**: `pullNextComponent` вызывался только после полного опустошения `pendingAspects`, но проверял `!pendingAspects.isEmpty()`. Ветка всегда была `false`.
- **Не было ledger'а исходных типов аспектов**: после обнуления аспекты удалялись из `pendingAspects` через `remove()`, теряя информацию об исходных типах.
- **Детерминированный выбор аспекта** вместо случайного.
- **`gateAllows` использовал `<=` вместо `<`**: при instability=0 gate позволял roll 0, тогда как оригинальный `if(instability > 0 && nextInt(500) < instability)` блокировал всё.
- **`isCosmeticSurge(20-20)`**: roll 0 — eject type 0, тест гарантированно падал.
- **`countAssignedRolls`** — тавтологичный счётчик через catch-all `isCosmeticSurge`, не проверял непересекаемость.
- **Статические contract-тесты** вместо поведенческих: 5 тестов v11.63.64 только дёргали захардкоженные `return true/false` в Parity-классе.

## Что сделано в v11.63.65

### 1. `lockedRecipeAspectTypes` ledger
Добавлено поле в `InfusionMatrixBlockEntity`:
```java
private final List<Aspect> lockedRecipeAspectTypes = new ArrayList<>();
```
- Заполняется при старте рецепта из `requiredAspects.keySet()` (все исходные типы).
- Сохраняется/загружается в NBT как `ListTag` с ключом `"LockedRecipeAspectTypes"`.
- Очищается в `clearCraftingState`, `activateMatrix`, `load()`.

### 2. Исправление refund-веток

**`pullNextComponent`**:
```java
// Было (недостижимо):
if (!pendingAspects.isEmpty() && ...) {
    Aspect extra = InfusionProcessHelper.firstPendingAspect(pendingAspects);
    pendingAspects.put(extra, pendingAspects.getOrDefault(extra, 0) + 1);
}
// Стало (достижимо, случайный выбор):
if (!lockedRecipeAspectTypes.isEmpty() && ...) {
    Aspect extra = lockedRecipeAspectTypes.get(level.random.nextInt(lockedRecipeAspectTypes.size()));
    pendingAspects.merge(extra, 1, Integer::sum);
}
```

**`drainEnchantmentXp`** — аналогично: случайный аспект из ledger вместо детерминированного порядка.

### 3. Исправление event-table контракта (`TC4InfusionInstabilityEventTableParity`)
- `gateAllows`: строгое `<` вместо `<=`, плюс проверка `instability > 0`.
- `countAssignedRolls`: считает категории на каждый roll и проверяет `categories == 1` (ровно одна категория).
- `CONTRACT_VERSION` обновлён до `"11.63.65"`.

### 4. Исправление GameTest-багов v11.63.63
- `isCosmeticSurge(20 - 20)` → `!isCosmeticSurge(0)` и `!isCosmeticSurge(20)` — roll 0 это eject type 0, roll 20 это warp.
- `gateAllows(0, 0)` → `!gateAllows(0, 0)` — при instability=0 ничто не проходит.
- `gateAllows(5, 5)` → `gateAllows(4, 5)` — строгое меньше.
- `countAssignedRolls` — коммент обновлён.

### 5. Новый поведенческий GameTest
Добавлен `infusionComponentShortageRefundReachesProductionPendingAspects` (72 тест):
- Проверяет, что production-константы (`TC4InfusionRuntime.componentShortageEssentiaRefundRollBound`) совпадают с parity-контрактом.
- Проверяет корректные значения roll-bound для индексов 0, 1, 9.
- Проверяет формулу instability-creep (100 и 50).
