# Thaumcraft Legacy Rebuild v11.63.98
## Research Mastery: атомарное объединение аспектов

**Дата:** 21.07.2026  
**Эталон:** Thaumcraft 4.2.3.5 для Minecraft 1.7.10  
**Целевая платформа:** Minecraft 1.19.2, Forge 43.5.2, Java 17

## Что делает оригинал

В `GuiResearchTable` при открытом исследовании `RESEARCHER2` игрок может зажать Shift и нажать на составной аспект. Клиент берёт два исходных компонента этого аспекта и отправляет обычный пакет объединения.

В `PacketAspectCombinationToServer` оригинал:

1. проверяет наличие обоих компонентов;
2. для каждого компонента сначала использует личный пул игрока;
3. бонус Research Table используется только тогда, когда личный пул этого компонента пуст;
4. списывает оба выбранных компонента до проверки результата;
5. поэтому невалидная, но полностью обеспеченная ресурсами пара всё равно расходует оба очка;
6. при нехватке любого компонента операция не должна начинаться.

Исходные места:

- `thaumcraft/client/gui/GuiResearchTable.java`, строки 820–833;
- `thaumcraft/common/lib/network/playerdata/PacketAspectCombinationToServer.java`, строки 82–117;
- `assets/thaumcraft/lang/en_US.lang`, ключ `tc.research_page.RESEARCHER2.1`.

## Что было в порте до v11.63.98

`ResearchTableFoundation.combine` выполнял два независимых вызова:

```java
consumePoolOrTableBonus(player, first)
consumePoolOrTableBonus(player, second)
```

Перед ними существовала проверка количества, поэтому обычная нехватка чаще всего отсеивалась заранее. Но списание не являлось одной транзакцией. Если второй adapter-вызов возвращал `false` из-за рассинхронизации меню, изменившегося bonus storage или другого состояния между preflight и commit, первый компонент уже был потерян.

Это особенно опасно для сочетаний источников `player pool + table bonus` и для одинаковых выбранных аспектов.

## Что сделано

### 1. Единый source-linked контракт

Добавлен:

```text
TC4ResearchMasteryCombinationParity
```

Он описывает:

- два компонента на одну попытку;
- приоритет личного пула перед бонусом таблицы;
- отдельный план источника для каждого списания;
- корректную обработку одинаковых аспектов;
- отказ preflight без частичного расхода;
- сохранение правила расхода компонентов у невалидной пары.

### 2. Атомарный production-путь

В `ResearchTableInventoryRuntime` добавлен:

```java
consumeAspectPairAtomically(player, first, second)
```

Операция:

1. снимает snapshot количества обоих аспектов в player pool и table bonus;
2. строит полный debit-plan без изменения данных;
3. отказывает до commit, если двух очков нет;
4. списывает запланированные значения;
5. при неожиданном отказе любого adapter-вызова восстанавливает точные snapshot-значения обоих источников.

`ResearchTableFoundation.combine` больше не выполняет два независимых списания и использует только транзакционный метод.

### 3. Точные transaction hooks

Добавлены внутренние методы для точного количества и rollback:

- `PlayerAspectKnowledge.setPoolAmountForTransaction`;
- `ResearchTableBlockEntity.consumeBonusAspect(aspect, amount)`;
- `ResearchTableBlockEntity.setBonusAmountForTransaction`;
- соответствующие делегаты в `ResearchTableMenu`.

### 4. Поведенческие GameTest

Добавлены четыре теста с реальными `Player`, `ResearchTableMenu` и `ResearchTableBlockEntity`:

1. **pool + table bonus** — Aer берётся из бонуса таблицы, Perditio из пула, создаётся Vacuos;
2. **нехватка второго компонента** — Aer остаётся нетронутым;
3. **невалидная пара** — Aer + Terra не создаёт аспект, но оба очка расходуются;
4. **одинаковый аспект** — один Aer недостаточен для пары Aer + Aer и не списывается частично.
