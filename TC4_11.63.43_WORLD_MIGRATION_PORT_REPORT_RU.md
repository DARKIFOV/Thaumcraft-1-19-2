# Thaumcraft Legacy Rebuild 1.19.2 — v11.63.43
## Отложенная миграция legacy-инвентарей и persistent chunk ledger

### Цель этапа

После закрытия общего registry fallback старые совместимые ID всё ещё могли оставаться в сохранённых сундуках, машинах, экипировке, эндер-сундуках и переносных инвентарях. До этого автоматическая замена выполнялась только для инвентаря игрока при входе и для появляющейся `ItemEntity`.

Этап 11.63.43 расширяет **только безопасный allow-list из 28 доказанных duplicate-ID**. Остальные legacy-ID не конвертируются автоматически: для материалов, recipe-компонентов и сохранённых migration-объектов точный registry ID остаётся частью совместимости.

## Реализовано

### 1. Сохранение полного ItemStack

`TC4LegacyDuplicateItemMigrator` больше не создаёт новый стек только из предмета, количества и обычного `tag`.

Теперь он:

1. сериализует исходный стек через `ItemStack#save`;
2. меняет только поле `id` на allow-listed canonical ID;
3. восстанавливает стек через `ItemStack#of`.

Так сохраняются количество, damage, custom name, enchantments, обычный NBT и сериализованный payload Forge capabilities.

### 2. Покрытые хранилища

Миграция выполняется для:

- основного инвентаря, брони и offhand игрока;
- эндер-сундука игрока;
- предметных сущностей;
- экипировки живых сущностей;
- entity-контейнеров, включая совместимые вагонетки;
- block entity, реализующих vanilla `Container`;
- block entity и entity с writable `IItemHandler`;
- вложенных переносных item-handler-инвентарей до глубины 4.

Если сторонний handler объявляет `IItemHandlerModifiable`, но отклоняет `setStackInSlot`, слот пропускается: старый мир не должен становиться незагружаемым из-за агрессивной миграции.

### 3. Хранилища без Container/capability

Добавлен интерфейс `TC4LegacyStackMigrationTarget`. Его реализуют:

- `ArcanePedestalBlockEntity` — установленный предмет;
- `ThaumatoriumBlockEntity` — catalyst;
- `InfusionMatrixBlockEntity` — travelling component, locked catalyst и failure snapshot.

### 4. Отложенный проход по чанкам

`ChunkEvent.Load` не выполняет полный обход содержимого. Он ставит загруженный `LevelChunk` в очередь.

В конце серверного тика уровня:

- обрабатывается не более **2 чанков**;
- используется `getChunkNow`;
- незагруженный к моменту обработки чанк пропускается;
- соседние чанки не загружаются принудительно;
- сканируются block entity и уже находящиеся внутри чанка живые/entity-контейнеры.

### 5. Persistent ledger

`TC4WorldMigrationSavedData` хранит данные отдельно для каждого измерения:

- schema `116343`;
- множество обработанных chunk-pos;
- число изменённых стеков;
- число block entity, в которых были изменения.

Изменения вызывают `setDirty`. При совпадающей схеме обработанные чанки не сканируются после перезапуска. Более старая/отсутствующая схема намеренно начинает новый ограниченный проход, поэтому будущий этап может безопасно повысить номер схемы.

1. `migration.loaded_chunk_container_sweep`;
2. `migration.player_ender_equipment_entities`;
3. `migration.nested_portable_item_handlers`;
4. `migration.chunk_budget_no_force_load`;
5. `migration.schema_bump_replay`.

До реального запуска Java 17 Forge остаются `NOT_TESTED`:

- реальные миры нескольких предыдущих версий;
- capability-NBT сторонних переносных контейнеров;
- сохранение после restart и повторная идемпотентность;
- нагрузка на server tick при массовой загрузке регионов;
- одновременный вход нескольких игроков;
- dedicated server и два клиента;
- handlers, которые имеют скрытые side-specific слоты.
