## 1. Результат нового compiler pass

GitHub Actions снова дошёл до настоящей задачи `:compileJava`. Предыдущая правка пакета `ServerLevelAccessor` сработала: соответствующих ошибок больше нет.

Компилятор показал одну независимую ошибку в `TC4WispEntity.java:248`:

```text
method checkMonsterSpawnRules in class Monster cannot be applied to given types
required: EntityType<? extends Monster>, ServerLevelAccessor, MobSpawnType, BlockPos, RandomSource
found:    ServerLevelAccessor, BlockPos, RandomSource
```

## 2. Корневая причина

В Wisp использовался недоступный для Mojmap 1.19.2 трёхаргументный вызов:

```java
Monster.checkMonsterSpawnRules(level, pos, random)
```

Метод `checkWispSpawnRules` уже принимает полный контекст spawn predicate, поэтому исправление не требует изменения поведения:

```java
Monster.checkMonsterSpawnRules(type, level, reason, pos, random)
```

Сохранены:

- общий `Mob.checkMobSpawnRules(type, level, reason, pos, random)`;
- радиус подсчёта соседних Wisp 16 блоков;
- ограничение оригинального порта `nearby < 8`;
- biome modifier, вес и размер группы;
- Wisp parity-контракты 11.62.87.

## 3. Защита от повторения

Добавлен `tools/tc4_116290_monster_spawn_rules_compile_guard.py`:

- запрещает трёхаргументный вызов;
- требует полный пятиаргументный Mojmap-контракт;
- проверяет сохранение пяти параметров spawn predicate;
- проверяет общий Mob rule;
- проверяет сохранение Wisp density cap;
- подключён в build/release workflow до Gradle.

Общий `forge_1192_compile_api_guard.py` также теперь запрещает известный ошибочный overload во всём Java-дереве.
