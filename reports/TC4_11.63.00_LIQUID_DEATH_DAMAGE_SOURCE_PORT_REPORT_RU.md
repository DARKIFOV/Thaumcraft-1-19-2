# Thaumcraft Legacy Rebuild — отчёт о переносе v11.63.00

## Цель стадии

Продолжить перенос Thaumcraft 4.2.3.5 с Minecraft 1.7.10 на Forge 1.19.2 и закрыть отсутствующий функциональный блок **Liquid Death / Жидкая смерть**, одновременно восстановив отдельные идентификаторы источников урона TC4.

## Перенесённая функциональность

### Liquid Death

- Зарегистрированы `FluidType`, source-fluid, flowing-fluid, `LiquidBlock` и настоящее ведро `thaumcraft:tc4_bucket_death`.
- Ведро заменяет прежний предмет-заглушку, поддерживает установку/забор жидкости и стандартное поведение раздатчика.
- Подключены byte-exact оригинальные `fluiddeath.png`, `fluiddeath.png.mcmeta` и `bucket_death.png`.
- Source и flowing fluid используют прозрачный render layer.
- Восстановлена наблюдаемая четырёхступенчатая шкала растворяющего урона: **4 / 3 / 2 / 1**.
- Добавлены фиолетовые пузырьки и звук `LAVA_POP` с исходной вероятностью 1/50, громкостью и диапазоном высоты тона.

### Источники урона TC4

Добавлен `TC4DamageSources`:

- `taint` — игнорирует броню и считается магическим уроном;
- `dissolve` — игнорирует броню;
- `tentacle` и `swarm` — сохраняют атакующую сущность для корректного сообщения о смерти.

Taint Poison, Taintacle, Giant Taintacle и Taint Swarm переведены с общих источников Minecraft на эти именованные источники.

### Кристаллы аспектов при растворении

Обработчик `LivingDropsEvent` восстанавливает ветку оригинального `EventHandlerEntity`:

1. Реагирует только на смерть от singleton-источника `dissolve`.
2. Получает аспекты погибшей сущности через существующий реестр аспектов.
3. Для каждого аспекта выполняет исходную 50-процентную ветку.
4. Использует исходную формулу количества: `max(1, (1 + nextInt(amount)) / 2)`.
5. Добавляет в список дропа кристалл, настроенный на соответствующий аспект.

## Адаптация API Forge 1.19.2

- Регистрация выполнена через существующие `DeferredRegister` и `RegistryObject`.
- Пара source/flowing связана через `ForgeFlowingFluid.Properties` вместе с bucket/block suppliers.
- Визуальные и физические свойства связаны через `FluidType` и `IClientFluidTypeExtensions`.
- Событие дропа обрабатывается на общем Forge event bus через `LivingDropsEvent`.

## Ограничения и честный статус

### Неэквивалентность конечной жидкости

Оригинал использовал `BlockFluidFinite` и четыре кванта. Стандартный `ForgeFlowingFluid` не является точной моделью сохранения конечного объёма. В порте сохранены видимая шкала урона и отсутствие преобразования потоков в новые источники, но точная finite-fluid mass conservation остаётся отклонением и требует отдельной реализации/тестирования.

### Компиляция и runtime

JAR не получен. Gradle Wrapper не дошёл до `compileJava`, поскольку среда не смогла разрешить `services.gradle.org` при загрузке Gradle 7.5.1 (`UnknownHostException`). В контейнере установлена Java 21, тогда как официальная документация Forge 1.19.x требует Java 17.

Следовательно, статический source/resource contract подтверждён, но клиентский рендеринг, сетевое поведение, фактический поток жидкости и игровой drop-runtime остаются **NOT TESTED** до успешной сборки и запуска на Java 17.
