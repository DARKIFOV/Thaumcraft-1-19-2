# Thaumcraft Legacy Rebuild v11.62.54 — единый отчёт

## 1. Статус ревизии

- **Целевая платформа:** Minecraft 1.19.2, Forge 43.5.2, Java 17.
- **Эталон поведения:** Thaumcraft 4.2.3.5 для Minecraft 1.7.10.
- **Базовая сборка:** v11.62.53.
- **Назначение v11.62.54:** восстановить оригинальный HUD жезла и исправить серверную смену фокусов так, чтобы порядок, дубликаты и невозможность сохранить старый фокус обрабатывались как в TC4.
- **Статус компиляции изменённых классов:** успешно, 0 ошибок.
- **Статус production-классов:** успешно reobf-нуты и проверены через `javap` на SRG Forge classpath.
- **Статус JAR:** создан patched-production JAR с полной повторной синхронизацией всех ресурсов из `src/main/resources`.
- **Runtime-статус:** v11.62.54 в графическом клиенте и на dedicated server ещё не запускалась. Визуальная и сетевая стабильность должна быть подтверждена в игре.

## 2. Что проверялось перед изменениями

### 2.1. Оригинальный HUD жезла

Эталон: `thaumcraft/client/lib/ClientTickEventsFML.java`, метод `renderCastingWandHud`.

Оригинальный TC4:

- рисует dial из `textures/gui/hud.png` в native-размере **32×32**;
- по умолчанию размещает его в левом верхнем углу;
- параметр `Config.dialBottom` переносит dial в левый нижний угол;
- при верхнем расположении дополнительно поворачивает шкалы на 90°;
- показывает шесть primal-vis резервуаров;
- показывает маркер стоимости активного фокуса;
- показывает маркер изменения vis относительно снимка раз в секунду;
- при зажатом Shift показывает целое количество vis и фактическую стоимость;
- рисует активный фокус в центре dial;
- для Equal Trade заменяет иконку фокуса выбранным блоком и показывает количество такого блока в инвентаре;
- показывает оставшийся cooldown фокуса в секундах.

### 2.2. Смена фокуса

Эталон: `thaumcraft/common/items/wands/WandManager.java`, метод `changeFocus`.

Ключевые контракты оригинала:

1. Используется настоящий `TreeMap<String, location>`.
2. Точные одинаковые sorting keys **перезаписывают** предыдущую запись, а не получают искусственный порядковый суффикс.
3. Сначала сканируются pouch-слоты Baubles; адаптером в 1.19.2 выступает pouch во второй руке.
4. Затем последовательно обходятся слоты инвентаря 0–35. В каждом слоте сразу обрабатывается прямой фокус или содержимое pouch.
5. Выбор выполняется как `exact key → higherKey → firstKey`.
6. Shift+F или отсутствие доступных фокусов снимает текущий фокус только в том случае, если его удалось сохранить.
7. При обычной смене новый фокус сначала извлекается из источника, но старый остаётся установленным, пока его не удалось безопасно вернуть в pouch или инвентарь.
8. Если старый фокус сохранить нельзя, выбранный новый фокус возвращается обратно, а жезл остаётся неизменным.

## 3. Найденные проблемы v11.62.53

### 3.1. HUD находился не там, где в TC4

Порт принудительно использовал:

```text
x = 8
y = screenHeight - 70
```

Это не соответствовало ни верхнему, ни нижнему варианту оригинального dial. Оригинальный default — `(0, 0)`; нижний вариант — `(0, screenHeight - 32)`.

### 3.2. Нижний dial всё равно использовал верхний поворот

Порт всегда применял дополнительный поворот на 90°. В TC4 он применяется только при `dialBottom == false`.

### 3.3. Отсутствовали центральная иконка и cooldown

В dial не рисовались:

- установленный фокус;
- выбранный Equal Trade блок;
- количество выбранного блока;
- оставшееся время cooldown.

### 3.4. Были добавлены неоригинальные текстовые строки

При Shift рядом с dial выводились суммарный vis и название фокуса. В оригинальном `renderCastingWandHud` этих двух строк нет; они меняли размеры и визуальный баланс HUD.

### 3.5. Значение vis отображалось не по оригинальной формуле

TC4 выводит:

```java
amt / 100
```

то есть целое значение vis. Порт использовал общий decimal formatter и мог показывать дробное значение там, где оригинал его не показывает.

### 3.6. Искусственные суффиксы ломали семантику TreeMap

Порт создавал ключи вида:

```text
AF...#0001
AF...#0002
```

Это давало два расхождения:

- точные дубликаты начинали участвовать в цикле отдельно, хотя оригинальный `TreeMap` оставляет только последнюю запись с одинаковым ключом;
- `foci.get(currentKey)` никогда не находил запись, поскольку реальный ключ содержал добавленный суффикс.

### 3.7. Смена фокуса была нетранзакционной

Старый фокус безусловно убирался с жезла, а при заполненном инвентаре мог быть выброшен в мир. Это не соответствует TC4: оригинал сохраняет старый фокус на жезле и возвращает выбранный фокус в источник, если swap невозможно завершить безопасно.

### 3.8. Shift+F воспроизводил звук даже без успешного изменения

Оригинальный звук `cameraticks` с pitch `0.9` воспроизводится только после успешного сохранения и снятия текущего фокуса.

## 4. Исправления HUD жезла

### 4.1. Оригинальная геометрия dial

Восстановлено:

```text
ширина: 32 px
высота: 32 px
верхний X: 0
верхний Y: 0
нижний Y: screenHeight - 32
```

`hud.png` по-прежнему содержит область 64×64, которая масштабируется в 0.5, как в TC4.

### 4.2. Перенесён `wand_dial_bottom`

В Forge-конфиг добавлен параметр:

```toml
[client_parity]
wandDialBottom = false
```

Default `false` соответствует `Config.dialBottom = false` в TC4 4.2.3.5.

При `false` шкалы получают оригинальный дополнительный поворот 90°. При `true` он не применяется.

### 4.3. Шесть primal reservoirs

Сохранены оригинальные atlas-координаты `hud.png`:

| Элемент | U | V | Размер |
|---|---:|---:|---:|
| Основа dial | 0 | 0 | 64×64, scale 0.5 |
| Цветная заливка | 104 | 0 | 8×30 max |
| Рамка резервуара | 72 | 0 | 16×42 |
| Маркер увеличения | 120 | 0 | 8×8 |
| Маркер уменьшения | 128 | 0 | 8×8 |
| Маркер стоимости | 136 | 0 | 8×8 |

Углы также сохранены:

```text
-15 + index × 24 градусов
```

### 4.4. Значения при Shift

Текущее количество снова выводится как:

```text
centivis / 100 с целочисленным делением
```

Стоимость фокуса показывает модифицированное значение с cap/gear discount и decimal formatting, как оригинальный `myFormatter`.

Удалены две неоригинальные строки:

- суммарный vis всех аспектов;
- название установленного фокуса.

### 4.5. Центральная иконка фокуса

Если установлен обычный фокус, его точный `ItemStack`, включая upgrades/NBT, рисуется в центре dial:

```text
x = dialX + 8
y = dialY + 8
```

Если legacy NBT содержит только тип, создаётся корректный fallback stack этого типа.

### 4.6. Equal Trade picked-block HUD

Если активен Equal Trade и выбран блок:

- в центре dial рисуется выбранный блок вместо иконки фокуса;
- подсчитываются совпадающие item IDs в инвентаре;
- количество выводится поверх иконки в half-scale, как `renderWandTradeHud`.

В 1.19.2 варианты старых metadata-блоков уже представлены отдельными item IDs или block states, поэтому сравнение `sameItem` сохраняет смысл оригинальной проверки item+metadata для переносимых блоков.

### 4.7. Cooldown

Для burst-фокусов берётся синхронизированный vanilla/Forge item cooldown:

```text
remainingSeconds = activationCooldownTicks × cooldownFraction / 20
```

Текст выводится в центре dial с масштабом 0.5 и суффиксом `s`.

### 4.8. Очистка client state

При `ClientPlayerNetworkEvent.LoggingOut` очищаются:

- все снимки старого vis по hotbar-слотам;
- флаги инициализации;
- таймер следующего снимка.

Это не позволяет стрелкам изменения vis переноситься между мирами или серверами.

## 5. Исправления смены фокусов

### 5.1. Удалены sequence suffixes

Удалена функция `uniqueKey` и формат `#%04d`.

Теперь используется ровно:

```java
foci.put(FocusPouchItem.sortingHelper(stack), location);
```

Таким образом, одинаковые ключи перезаписываются в том же стиле, что и в оригинальном `TreeMap`.

### 5.2. Восстановлен порядок сканирования

Новый порядок:

1. pouch во второй руке как адаптер Baubles pouch;
2. инвентарь 0–35;
3. в каждом слоте сначала прямой focus, затем содержимое pouch этого слота.

Поздняя запись с тем же sorting key заменяет раннюю, как в TC4.

### 5.3. Восстановлен выбор exact → higher → wrap

Алгоритм:

```text
location = foci.get(requestedKey)
если нет: location = foci.higherEntry(requestedKey)
если нет: location = foci.firstEntry()
```

Это исправляет циклическое переключение F и поведение одинаковых фокусов/upgrade-комбинаций.

### 5.4. Транзакционная смена

Новый swap:

1. Выбранный focus извлекается из конкретного inventory/pouch slot.
2. Установленный focus пытается сохраниться:
   - во второй-hand pouch;
   - в inventory pouches по порядку;
   - в обычный инвентарь.
3. Только после успешного сохранения старого focus очищается с жезла.
4. Новый focus устанавливается на жезл.
5. Если старый focus сохранить нельзя, новый возвращается в **точно тот же slot**, который был освобождён.
6. Если точный rollback невозможен из-за внешнего изменения инвентаря в тот же момент, используется общий безопасный storage fallback; drop остаётся только последней защитой от удаления уже извлечённого предмета.

При нормальном серверном tick exact rollback всегда должен попасть в только что освобождённый slot.

### 5.5. Безопасное Shift+F

Текущий focus снимается только после успешного сохранения. Если pouch и инвентарь полностью заполнены:

- focus остаётся установленным;
- предмет не выбрасывается;
- звук смены не проигрывается.

### 5.6. Удалены неоригинальные action-bar сообщения F-переключения

Оригинальный `WandManager.changeFocus` сообщает о смене звуком `cameraticks`, а не строками `Equipped ...` / `No wand foci found`.

Сообщения отдельного right-click адаптера pouch сохранены: это не часть F-пакета и служит объяснением modern off-hand управления.

## 7. Сборка и аудит JAR

Полный Gradle Wrapper в этой среде всё ещё не может загрузить Gradle 7.5.1 из `services.gradle.org`. Поэтому использован контролируемый patched-production pipeline на базе v11.62.53.

В отличие от минимальной замены отдельных entries, pipeline:

1. удаляет из базового JAR все управляемые `assets/`, `data/`, `pack.mcmeta`, `mods.toml` и access transformer;
2. повторно добавляет **все 6120 ресурсов** из текущего `src/main/resources`;
3. заменяет 7 новых reobf-class files;
4. создаёт новый manifest с `Implementation-Version: 11.62.54`;
5. проверяет итоговый архив через `unzip -t`;
6. сравнивает все ресурсы и основные классы отдельным `audit_release_jar.py`.

## 10. Runtime-проверка v11.62.54

### P0 — HUD

1. Взять жезл без фокуса: dial должен быть 32×32 в левом верхнем углу.
2. Проверить `wandDialBottom=true`: dial должен переместиться ровно в левый нижний угол и изменить ориентацию шкал.
3. Проверить GUI Scale 1/2/3/Auto.
4. Зарядить и разрядить каждый primal: цветная шкала и стрелки должны совпадать с TC4.
5. Зажать Shift:
   - слева/снаружи должно показываться целое количество vis;
   - стоимость должна отображаться только у нужных аспектов;
   - не должно быть дополнительных строк total/focus.
6. Установить каждый focus: иконка должна находиться в центре dial.
7. Использовать burst-focus: countdown должен уменьшаться плавно и исчезать после cooldown.
8. Equal Trade:
   - выбрать блок;
   - убедиться, что в центре отображается блок, а не иконка focus;
   - проверить корректность количества в инвентаре.
9. Перезайти в мир и проверить отсутствие ложных стрелок изменения vis.

### P0 — переключение F

1. Положить несколько разных фокусов в инвентарь и pouch.
2. Проверить сортировку и wrap по F.
3. Проверить Shift+F.
4. Проверить одинаковые фокусы с одинаковыми upgrades: они не должны образовывать искусственную отдельную цепочку из sequence suffixes.
5. Проверить одинаковый тип с разными upgrades: sorting helper должен различать их по upgrade NBT.
6. Полностью заполнить pouch и инвентарь, затем:
   - попытаться снять фокус;
   - попытаться сменить фокус;
   - убедиться, что установленный focus не выпадает и не исчезает.
7. Проверить pouch во второй руке и pouch в обычном инвентаре.
8. Повторить на dedicated server с сетевой задержкой.

## 12. Итог

v11.62.54 продолжает точечную реконструкцию по исходникам TC4, а не косметическое приближение. В этой ревизии восстановлен настоящий 32×32 wand vis dial, его позиционирование, ориентация, central focus/trade icon, cooldown и Shift-значения. Одновременно исправлен более опасный серверный дефект: смена фокуса теперь снова использует оригинальную `TreeMap`-семантику и не может выбросить или потерять установленный focus из-за заполненного инвентаря.

Ревизия прошла компиляцию изменённых классов, reobfuscation, статические guards, ресурсные/model-аудиты и полный release-JAR audit. До запуска в настоящем Forge-клиенте и на dedicated server она имеет статус **статически проверенной, но не runtime-подтверждённой**.
