# Отчёт v11.63.97 — поведенческая parity Research Table

## Эталон TC4 4.2.3.5

В `TileResearchTable` исходной версии:

- серверный счётчик проверяется выражением `nextRecalc++ > 600`;
- значение 600 ещё не запускает пересчёт и становится 601;
- значение 601 запускает пересчёт и сбрасывается в 0;
- при сохранении `bonusAspects` записывается максимум одна NBT-запись на каждый положительный тип аспекта, независимо от его количества в памяти;
- при загрузке каждая запись восстанавливает одну единицу аспекта.

## Ошибка порта до v11.63.97

Порт записывал одну NBT-запись на каждую единицу бонусного аспекта. Например, `Aer:3` переживал перезапуск как `Aer:3`, тогда как оригинальный TC4 после сохранения/загрузки восстанавливал `Aer:1`. Это создавало накопление бонусов, которого не было в оригинальном persistent-состоянии.

## Что исправлено

1. Добавлен source-linked контракт `TC4ResearchTableBehaviorParity`.
2. `ResearchTableBlockEntity.serverTick` использует точную post-increment семантику оригинала.
3. NBT-сохранение пишет одну запись на положительный тип аспекта.
4. NBT-загрузка нормализует дубли из старых сохранений порта до одной единицы на тип.
5. После каждого сработавшего пересчёта таблица теперь безусловно помечается изменённой и синхронизируется, даже если RNG не добавил аспект: сброшенный счётчик должен сохраниться.
6. Добавлены три настоящих GameTest, работающих с реальным `ResearchTableBlockEntity`:
   - сжатие дублирующихся количеств и нормализация legacy-дублей при NBT round-trip;
   - сохранение результата потребления бонуса;
   - граница счётчика 600/601 через реальный `serverTick`.

## Проверяемые показатели

- 107 уникальных `@GameTest` методов;
- 435 уникальных сценариев runtime manifest;
- 258/258 оригинальных recipe registrations статически mapped;

## Ограничение

Статический guard и Java parse не заменяют Forge runtime. Для статуса runtime PASS необходимы Java 17, успешный `gradlew build`, JAR с SHA-256 и реальный `runGameTestServer` для всех 107 тестов.
