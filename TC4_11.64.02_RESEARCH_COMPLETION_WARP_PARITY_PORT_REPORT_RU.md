# Thaumcraft Legacy Rebuild v11.64.02 — Research Completion Warp Parity

## Цель раунда

Исправить распределение warp, которое выполняется при завершении исследования. В предыдущем порте весь `ResearchEntry.warp()` записывался как permanent warp.

## Поведение оригинального Thaumcraft 4.2.3.5

`ResearchManager.completeResearch`, строки 611–627:

- `warp == 0` — ничего не добавляется;
- `warp == 1` — одна единица permanent warp;
- `warp > 1` — `sticky = warp / 2`, `permanent = warp - sticky`;
- temporary warp при завершении исследования не используется;
- повторное завершение уже открытого исследования не начисляет warp повторно.

Примеры: `2 → permanent 1 + sticky 1`, `3 → 2 + 1`, `5 → 3 + 2`, `6 → 3 + 3`.

## Ошибка порта

`OriginalResearchProgression.applyUnlockSideEffects` выполнял:

```java
PlayerThaumData.addWarpPermanent(player, warp);
```

Из-за этого sticky warp исчезал как отдельная категория. Это влияло на очистку sticky warp, счётчики событий и диагностику общего/actual warp.

В полном индексе из 201 исследований warp имеют 20 записей. Для 12 записей warp выше 1, поэтому ошибка меняла фактическое распределение: `FOCUSHELLBAT`, `SINSTONE`, `INFERNALFURNACE`, `JARBRAIN`, `LIQUIDDEATH`, `BOTTLETAINT`, `GOLEMFLESH`, `ADVANCEDGOLEM`, `CRIMSON`, `OCULUS`, `FOCUSPRIMAL`, `ROD_primal_staff`.

## Реализация v11.64.02

Добавлен единый source-linked контракт:

```text
TC4ResearchCompletionWarpParity
```

Production теперь получает `WarpSplit` и отдельно вызывает:

```java
PlayerThaumData.addWarpPermanent(player, split.permanent());
PlayerThaumData.addWarpSticky(player, split.sticky());
```

`WarpCounter` после обоих вызовов остаётся равен общей сумме permanent + sticky + temporary.
